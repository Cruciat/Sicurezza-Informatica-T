# REPORT ESERCIZIO WEB SECURITY (SQLi)

## 1. Logica e Progettazione delle Query (Exploitation)
- **Fase di Discovery:** Visitando l'applicazione web e manipolando il parametro in GET, è stata tentata un'iniezione Error/Boolean-based classica:
  `?id=Alice'+OR+'1'='1`
  Questa operazione ha bypassato i filtri originali, rivelando tutti gli utenti presenti nel database e mostrando un messaggio nascosto che indicava l'uso di SQLite.
  
- **Determinazione delle Colonne (UNION Based):** Tramite la tecnica dell'iniezione di valori `NULL`, è stato determinato che la query di backend si aspetta esattamente [INSERIRE NUMERO, es. 4] colonne:
  `?id=Alice' UNION SELECT NULL,NULL,NULL,NULL --`

- **Enumerazione dello Schema (SQLite):**
  È stata interrogata la tabella di sistema `sqlite_schema` per estrarre i nomi delle tabelle presenti nel database:
  `?id=Alice' UNION SELECT type, name, tbl_name, NULL FROM sqlite_schema --`
  Sono emerse le seguenti tabelle interessanti: [INSERIRE TABELLE, es. flags, people].

- **Esfiltrazione Dati:**
  Individuata la tabella target, è stata effettuata una query diretta per estrarne il contenuto:
  `?id=Alice' UNION SELECT * FROM [NOME_TABELLA] --`

## 3. Vulnerabilità e Mitigazione
L'errore di progettazione risiede nell'assenza di sanitizzazione e validazione dell'input utente. Il parametro in GET viene concatenato direttamente all'interno della query SQL, permettendo all'attaccante di alterarne la logica sintattica.

**Risoluzione:**
1. **Prepared Statements:** Sostituire la concatenazione di stringhe con l'uso di query parametrizzate (es. istruzioni `PREPARE`), affinché l'input venga trattato dal database esclusivamente come dato e non come codice eseguibile.
2. **Validazione (Allow-listing):** Laddove possibile, implementare un controllo che accetti l'input solo se corrisponde a una lista di valori noti e considerati sicuri (es. solo caratteri alfanumerici specifici).
